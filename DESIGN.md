# gochickpeas — design

Go implementation of [RustyChickpeas](https://github.com/freeeve/rustychickpeas):
a high-performance in-memory graph database. Reads and writes RCPG files
byte-compatibly with the Rust implementation and targets feature parity with
`rustychickpeas-core`. Module `github.com/freeeve/gochickpeas`, root package
`chickpeas`.

## Scope

- **In scope:** the engine (Builder -> immutable Snapshot -> Manager), RCPG/RRSR
  codecs, traversal + search + aggregation kernels, analytics (WCC/PageRank/
  CDLP/LCC/SSSP), full-text (BM25) and geo (k-d tree) indexes.
- **Out of scope:** CSV/Parquet/S3 bulk loading--RCPG files are the interchange
  format (loading stays a Rust-side capability); Cypher (future package, see
  seam note below).
- **Terminology:** graph relationships are "rel(s)", never "edge(s)".

## Package layout

```
github.com/freeeve/gochickpeas      go 1.24; single dep: RoaringBitmap/roaring/v2
├── rcpg/                           RCPG codec (extractable subtree; imports only roaring+stdlib)
│   ├── rcpg.go section.go column.go cursor.go lazy.go errors.go
│   ├── cmd/gencorpus/              writes the corpus with Go builders (reverse interop)
│   ├── internal/conformance/       Go twins of the Rust corpus builders
│   └── testdata/conformance/       golden corpus generated by rustychickpeas-format
├── rcpg/rrsr/                      RRSR record store (roaringrange RECORDS.md-compatible)
├── nodeset/                        Set (roaring-backed) + ParFold          [M3]
├── internal/parallel/              For/Fold/Join chunked worker pools      [M3]
├── internal/bitset/                []uint64 bitset                         [M3]
├── (root) package chickpeas        the engine, one package, small files    [M3+]
└── cypher/                         FUTURE — reserved path
```

Rules: `rcpg` never imports `internal/*` from the engine, so it can be
extracted to its own repo/module by `git mv`. Files stay under 500 lines
(1000 worst case). Dense-bool column data stays raw LSB-first packed bytes
inside `rcpg`; the engine converts to its own bitset when materializing.

## Byte compatibility

The corpus in `rcpg/testdata/conformance/` is generated by
`rustychickpeas-format`'s `gen_conformance` example (the Rust repo owns
FORMAT.md and is the source of truth). Guarantees, enforced by tests:

1. Every positive file parses to the manifest's facts (counts, adjacency and
   column spot checks).
2. Re-encoding a parsed file reproduces the golden bytes exactly.
3. Rebuilding each logical graph from scratch with Go builders also
   reproduces the golden bytes (covers array/bitmap/run roaring containers,
   NaN payloads, -0.0, LSB-first unaligned dense bool, atom-0 missing
   strings, sparse IDs, `in_to_out` pairing hazards).
4. The reverse direction: `go run ./rcpg/cmd/gencorpus <dir>` then
   `RCPG_INTEROP_CORPUS=<dir> cargo test -p rustychickpeas-format --test
   go_interop` proves Rust parses Go-written files to bit-exact equality.
5. Negative corpus files fail with typed errors (`ErrCorrupt`,
   `ErrUnsupportedVersion`), never a panic; fuzz targets keep it that way.

Lockstep rule: any FORMAT.md change regenerates the corpus in paired commits
(Rust generator + this repo's vendored copy). `rcpg.Version` pins v1.

Cross-language subtleties already encoded in the corpus: Go `math.NaN()` is
`0x7FF8...0001` while Rust `f64::NAN` is `0x7FF8...0000`; Go roaring's
`AddRange` builds run containers where Rust's per-id insert builds
array/bitmap containers--byte identity requires matching container choices.
Eager `Parse` bounds-checks every directory entry while `ParseLazy` never
fetches unwanted/unknown sections (same asymmetry as the Rust codec).

## API conventions (engine, M3+)

- Go-idiomatic short names: `Snapshot`, `Builder`, `Manager`; `Rel` in
  compounds (`AddRel`, `RelType`, `RelRef`, `RelCount`).
- Traversal returns `iter.Seq` (range-over-func): `Neighbors(n, dir,
  relTypes ...string) iter.Seq[NodeID]`; hot paths take a pre-resolved
  `RelMatch` via `*Match` method siblings.
- `Value` is a comparable `struct{kind ValueKind; bits uint64}` (map-key
  semantics identical to Rust's `ValueId`; F64 compared by bit pattern).
- `Prop`'s zero value means absent, folding Rust's `PropExt` into one type:
  `g.Prop(n, "age").I64Or(0)`.
- `(T, error)` with `errors.Is`-able sentinels for constructors/codecs;
  comma-ok for pure lookups.
- Convenience polymorphism at the boundary (`NodesWithProperty(label, key,
  value any)`) over a typed core (`NodesWithValue`).
- Parallel kernels use `internal/parallel.Fold` (per-chunk accumulators,
  chunks = workers x 4, mirroring rayon's fold/reduce shape); public
  signatures stay pure std (`func` values only). Lazy indexes:
  build-outside-lock, discard racing duplicate; `sync.OnceValue` for stats.
- NodeSet is roaring-only behind a private repr; the Rust adaptive
  small-set arm lands later only if Go benchmarks prove the win.

## Cypher seam (future)

The Rust cypher engine runs backend-generic over a `CypherGraph` trait; the
future Go `cypher` package will define its own consumer-side interface over
`*Snapshot` (Go idiom: interfaces live with the consumer). The engine must
therefore keep its full read surface as methods on `*Snapshot`--schema
introspection, atom/string resolution both directions, columnar access
including rel columns by CSR position, `RelMatch` traversal, lazy indexes--
never trapped in free functions an interface couldn't capture. The engine's
4-kind `Value` stays columnar; richer runtime values (lists/maps/temporals)
belong to the cypher package.

## Milestones

Task files live in `tasks/` (Eve's numbered convention; rename to
`.done.md` when complete).

| # | Milestone | Status |
|---|-----------|--------|
| M0 | Scaffold: go.mod, CI, DESIGN.md, PARITY.md generator | done |
| M1 | rcpg + rrsr codec, conformance both directions, fuzz | done |
| M2 | Core types: NodeID/Direction/Value, Atoms, bitset, nodeset.Set, internal/parallel | done |
| M3 | Snapshot read path: GraphSection bridge + inToOut, RelMatch, iterators, Col/Prop, lazy prop indexes | done |
| M4 | Builder + Finalize: AddNode/AddRel, parallel CSR, dense/sparse/rank column selection (exact Rust thresholds) | done |
| M5 | Search kernels: BFS x3, CanReach, Neighborhood, Dijkstra, WeightedShortestPath | done |
| M6 | Aggregation kernels: RootsVia/NeighborVia/FoldVia/CoOccurring/CommonNeighborCounts/NeighborGroups + fluent Aggregation | done |
| M7 | Analytics: WCC/PageRank/CDLP/LCC/SSSP | done |
| M8 | Full-text (BM25) + geo (k-d tree) | done |
| M9 | Manager, benchmarks, v0.1.0 | done |

Known deviations from Rust (deliberate, noted per-row in PARITY.md as it
fills in): loader/dedup surfaces are out of scope (RCPG is the
interchange); unknown rel-type names uniformly match nothing (Rust's
can_reach treats all-unknown as match-all); the eager Finalize property
index builds from finished columns (identical to the lazy path) rather
than staged pairs. The type-index divergence originally found here
(builder index vs FORMAT.md's outgoing-CSR positions) was fixed in Rust
core (rustychickpeas task 196, commit 96243bb) -- both builders now key
by CSR position.

Parity is tracked in `PARITY.md`, generated by `scripts/parity.sh` from the
Rust repo's public API (rows: rust symbol / go symbol / status / notes;
`wontport` rows carry a reason).

## Testing

- `go test ./... -race`; coverage target >80% overall, ~95% for `rcpg`.
- Fuzz targets (corpus-seeded, run briefly in CI): `FuzzParse`,
  `FuzzParseLazy`, rrsr `FuzzParse`.
- Behavioral suites port `graph_snapshot/tests.rs` and friends as
  table-driven tests per milestone.
- Benchmarks mirror the criterion suites with `testing.B.Loop`; Go-vs-Go
  regressions tracked in CI, Rust-vs-Go comparisons documented but never
  gated.
